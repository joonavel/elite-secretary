# Requirements Specification

## 0. 문서 정보
- 프로젝트명: **Elite Secretary — 온라인 회의 녹음 → Action Item 자동 추출 및 처리 Agent**
- 문서 목적: SDD(Software-Driven Development) 1단계 요구사항 정의
- 작성 범위: 전체 제품 비전 + 해커톤 데모 시나리오 요구사항

## 1. Problem Statement (문제 정의)

조직 내 업무 요청의 상당수는 회의에서 구두로 전달되며, 회의가 끝난 뒤 담당자는 아래 수작업을 반복적으로 수행해야 한다.

- 회의 내용 수동 정리·요약
- Action Item 파악 및 담당자 이관
- 후속 일정 조율 및 캘린더 등록
- 필요 자료 조사·데이터 취합 (Excel, 보고서 등)
- 보고서·문서 초안 작성

본 프로젝트는 **Teams 회의 녹음 파일**을 입력으로 받아, 회의 종료 후 **Action Item을 자동으로 추출하고 유형에 따라 적합한 Agent가 후속 처리(일정 예약·템플릿 생성·자료 조사)를 자동 수행**하는 End-to-End Agent 워크플로우를 구현한다.

해커톤 제약으로 모든 Action Item 유형을 구현하기 어렵기 때문에, **"자료 조사 → 재무 문서 작성"** 시나리오 하나를 완전히 구현하여 아키텍처의 확장 가능성을 증명한다.

## 2. Goals (목표)

### 2.1 핵심 목표
1. **회의 종료 후 자동 실행**: 추가 사용자 입력 없이 Action Item 처리
2. **Action Item 유형별 라우팅**: STT → 요약/추출 → 유형 판별 → 전문 Agent 실행
3. **수치 기반 산출물 생성** (데모 시나리오): 숫자·함수·그래프 포함 Excel + Insight 문서
4. **확장 가능성 시연**: 동일 구조로 일정 예약·템플릿 생성 등 타 Action Item 유형으로 확장 가능함을 증명

### 2.2 성공 기준 (Success Criteria)
- 데모 시나리오(LLM 비용 분석)에서 End-to-End 실제 실행 가능
- 상반기(H1) 기준 LLM 비용/토큰 사용량을 **팀별 집계**
- **Excel 보고서 + Insight 문서** 자동 생성

### 2.3 보조 목표
- 1개 이상의 시각화 그래프를 포함
- 미구현 Action Item 유형(일정 예약, 템플릿 생성)의 확장 구조를 설계 수준에서 제시

## 3. Scope

### 3.1 In Scope
- Teams 채팅에서 회의 녹음 파일 수집
- 녹음 파일을 Azure Speech SDK(File Input)로 STT 처리
- 회의 텍스트 요약 및 Action Item 구조화 추출
- Action Item 유형 판별 및 Agent 라우팅 설계
- **[데모 시나리오]** 재무 데이터(Excel) 조회·집계·보고서 생성·Insight 문서 작성

### 3.2 Out of Scope (해커톤 범위)
- 일정 예약 Agent 실제 구현 (Calendar API 연동)
- 템플릿 생성 Agent 실제 구현
- Teams 전사(Transcript) 기능 직접 활용
- Human-in-the-loop 승인 프로세스
- 복잡한 예산 예측/시뮬레이션 모델링
- 다국어 회의 지원

## 4. Design Decisions (중요 설계 결정)

### 4.1 단일 시나리오 집중 전략
- 해커톤 시간 제약으로 모든 Action Item 유형을 구현하지 않는다.
- **"자료 조사 → 재무 문서 작성"** 시나리오를 완전히 구현하여 동작을 증명한다.
- 라우팅 구조는 설계 수준에서 확장 가능하게 유지하여 다른 유형 추가를 쉽게 한다.

### 4.2 STT 방식
- 선택: **Azure Speech SDK – File Input**
- 이유:
  - 입력이 이미 존재하는 녹음 파일
  - Batch Job/Storage 관리 복잡성 최소화
  - Agent 본연 역할(판단/실행)과 책임 분리 명확화

### 4.3 책임 분리 원칙
- STT는 **전처리 단계**
- Action Item 추출·유형 판별·Agent 라우팅·도구 실행·산출물 생성은 **Agent 책임**

### 4.4 데이터 원천 결정 (데모 시나리오)
- 비용/토큰 집계의 공식 원천은 **사내 재무 Excel(수동 업데이트 원본)** 으로 고정

### 4.5 화자 분리 정책
- Diarization은 **best-effort + feature flag(기본 on)** 으로 적용한다.

## 5. Use Case 시나리오

### 5.1 전체 제품 비전 — Action Item 유형별 처리

회의에서 발생하는 Action Item은 아래 세 가지 유형으로 분류되며, 각 유형에 맞는 Agent가 자동 실행된다.

| Action Item 유형 | 예시 발화 | 처리 Agent |
|---|---|---|
| 📅 일정 예약 | "다음 주 화요일에 팀 전체 미팅 잡아줘" | Calendar Agent |
| 📝 템플릿 생성 | "이번 프로젝트 킥오프 문서 초안 만들어줘" | Template Agent |
| 🔍 자료 조사 / 문서 작성 | "상반기 LLM 비용 정리해서 보고서 써줘" | Research + Report Agent |

### 5.2 해커톤 데모 시나리오 — LLM 비용 분석 보고서

**회의 상황**: 팀장급이 재무담당자에게 요청
> "올해 상반기 전사 LLM 토큰 사용량과 비용을 정리해 주세요."

**자동 처리 흐름**:
1. Teams 녹음 파일 수집
2. STT로 회의 텍스트화
3. Agent A: 회의 요약 + Action Item 추출 → "자료 조사·문서 작성" 유형으로 분류
4. Agent B: 사내 재무 Excel에서 팀별/월별 토큰·비용 데이터 조회·집계
5. Agent C / D (병렬): Excel 리포트 생성 + Insight 문서 생성

**기대 결과물**:
1. **재무 분석 Excel**
   - 팀별 / 월별 토큰·비용
   - 합계, 평균 등 기본 함수 포함
   - 팀별 비용 비교 그래프
2. **Insight 문서**
   - 수치 기반 인사이트 (예: 개발팀 비용 편중, 팀별 할당량 조절 권고)
   - 비용 편중·비효율 패턴 분석
   - 실행 가능한 권고안 제시

## 6. Functional Requirements

### FR-1 녹음 파일 수집
- FR-1.1 Teams 채팅에서 회의 녹음 파일을 가져올 수 있어야 한다.
- FR-1.2 파일 포맷은 mp4 / m4a 등 Teams 기본 녹음 포맷을 지원해야 한다.
- FR-1.3 분석 대상 회의를 식별할 수 있는 최소 메타데이터(회의명, 시간, 채팅 링크 또는 파일 ID)를 저장해야 한다.

### FR-2 Speech to Text (STT)
- FR-2.1 녹음 파일을 Azure Speech SDK(File Input)로 텍스트화해야 한다.
- FR-2.2 화자 분리는 **feature flag로 제어**할 수 있어야 하며, 기본값은 on 이어야 한다.
- FR-2.3 화자 분리는 best-effort로 처리하며, 미지원/실패 시 해당 사실을 로그에 남기고 STT 본 처리는 계속되어야 한다.
- FR-2.4 도메인 용어(LLM, token, cost 등) Phrase List 보정 기능을 제공해야 한다.
- FR-2.5 STT 결과(원문 텍스트, 타임스탬프 등)를 후속 처리 가능한 구조로 저장해야 한다.

### FR-3 회의 요약 및 Action Item 추출 - Agent A
- FR-3.1 회의 텍스트를 요약해야 한다.
- FR-3.2 Action Item을 구조화된 형태(예: 유형, 담당자, 기한, 세부 내용)로 추출해야 한다.
- FR-3.3 Action Item 유형(일정 예약 / 템플릿 생성 / 자료 조사·문서 작성)을 판별해야 한다.
- FR-3.4 추출 결과는 후속 Agent 라우팅에 활용 가능한 구조화된 형태여야 한다.

### FR-4 재무 데이터 조회 - Agent B (데모 시나리오)
- FR-4.1 사내 재무 Excel(수동 업데이트 원본)에서 팀별/월별 토큰 및 비용 데이터를 조회할 수 있어야 한다.
- FR-4.2 조회 대상 기간은 기본적으로 H1(1~6월)이며, Action Item 추출 결과로 변경 가능해야 한다.
- FR-4.3 누락 값/형식 오류(숫자 아님 등) 탐지 시 오류를 명시적으로 보고해야 한다.
- FR-4.4 데이터 검증 결과는 `warning`과 `error`로 구분되어야 한다.

### FR-5 집계 및 계산 - Agent B (데모 시나리오)
- FR-5.1 팀별·월별 토큰/비용 집계를 수행해야 한다.
- FR-5.2 총합, 평균 등 기본 계산값을 산출해야 한다.
- FR-5.3 집계 로직은 재현 가능해야 하며 동일 입력에 동일 결과를 보장해야 한다.

### FR-6 Excel 결과물 생성 - Agent C (데모 시나리오)
- FR-6.1 결과 Excel 파일을 자동 생성해야 한다.
- FR-6.2 최소 1개 이상의 시각화 그래프(예: 팀별 비용 비교)를 포함해야 한다.
- FR-6.3 보고서에는 산출 기준(기간, 데이터 원천, 생성 시각)을 메타정보로 포함해야 한다.

### FR-7 Insight 문서 생성 - Agent D (데모 시나리오)
- FR-7.1 집계 수치를 근거로 핵심 인사이트를 자동 작성해야 한다.
- FR-7.2 비용 편중/비효율 패턴을 설명해야 한다.
- FR-7.3 실행 가능한 간단 권고안을 포함해야 한다.

### FR-8 오케스트레이션/자동 실행
- FR-8.1 회의 종료 이벤트(또는 데모용 트리거) 이후 파이프라인이 자동 실행되어야 한다.
- FR-8.2 수집 → STT → Action Item 추출 → 유형별 Agent 실행 → 결과물 생성이 순차 실행되어야 한다.
- FR-8.3 각 단계 상태(성공/실패/진행중)를 추적 가능해야 한다.

### FR-9 로그 및 추적성
- FR-9.1 단계별 입력/출력 요약 로그를 남겨야 한다.
- FR-9.2 실패 시 실패 단계와 원인을 식별 가능하게 기록해야 한다.

### FR-10 오류 처리
- FR-10.1 파일 없음, STT 실패, 데이터 누락 등 주요 오류 케이스를 명시적으로 처리해야 한다.
- FR-10.2 실패 시 부분 결과물 생성 여부와 재실행 가능성을 명확히 해야 한다.
- FR-10.3 검증 `error` 발생 시 파이프라인을 중단하고, 검증 `warning`만 존재하면 후속 단계를 계속 진행해야 한다.

### FR-11 특이사항
- FR-6과 FR-7을 담당하는 Agent들은 병렬적으로 진행하는 것으로 한다.

## 7. Non-Functional Requirements

### NFR-1 데모 실행성
- 단일 데모 세션에서 End-to-End 실행이 가능해야 한다.

### NFR-2 신뢰성
- 동일 입력 기준 결과 재현성이 있어야 한다.

### NFR-3 성능(해커톤 기준)
- 10분 길이 회의 녹음 입력 기준, 전체 파이프라인은 **10분 이내**에 완료되어야 한다(외부 API 지연 시간 제외).

### NFR-4 유지보수성·확장성
- STT / Action Item 추출 / 라우팅 / 유형별 Agent 모듈이 분리되어 새로운 Action Item 유형을 추가할 수 있는 구조여야 한다.

## 8. Assumptions & Constraints
- 사내 재무 Excel은 최신 데이터가 반영된 신뢰 가능한 원천이라고 가정한다.
- 인증/권한(Teams 파일 접근, Speech API 키 등)은 사전 준비된 상태를 전제로 한다.
- 본 단계에서는 단일 언어(한국어 또는 프로젝트 기본 언어) 회의를 우선 지원한다.
- 해커톤 범위에서는 Action Item 유형이 1개(자료 조사·문서 작성)만 완전 구현된다.

## 9. Acceptance Criteria
- AC-1 지정된 회의 녹음 파일 1건 입력 시 데모 시나리오 Step 전체가 성공하고 최종 상태가 `SUCCEEDED`로 기록된다.
- AC-2 팀별/월별 토큰 및 비용 집계가 포함된 Excel이 생성된다.
- AC-3 Excel에는 최소 1개 그래프가 포함된다.
- AC-4 Insight 문서에는 최소 2개 이상의 인사이트가 포함되고, 각 인사이트는 최소 1개 이상의 수치 근거를 포함한다.
- AC-5 실행 로그로 단계별 성공/실패를 확인할 수 있다.
- AC-6 시스템 아키텍처가 다른 Action Item 유형(일정 예약, 템플릿 생성)의 추가를 지원할 수 있는 구조임을 문서로 확인할 수 있다.
